# v3.3.0 — Rationale & decision log (the *why* behind the *what*)

## Goal & invariants

- **Why the parity rule (every capability in /admin AND the GPT).** Every time the design optimized for "how Emy specifically will use it," it doubled back — the clearest case was the by-link image share, which had to be rebuilt as a real attach/upload. Designing for the reusable-template **"User"** (a larger catalog, either surface possibly down) is what stops that rework. A Custom GPT outage must never strand the owner, so /admin has to be complete as if the GPT didn't exist, and vice versa.
- **Why single-admin is an explicit scope boundary, not a gap.** Simultaneous multi-admin / multi-seat touches far more than any one race (read-then-write everywhere), so it's a future formal, through-and-through effort — never patched piecemeal here. The clean follow-up *if* multi-seat ever ships is an atomic `record_relist` RPC (the relist read-modify-write is the one place a second operator could race); deliberately deferred, not overlooked.

## WS1 — Refund

- **Why amount-based, not "refund the order."** A Stripe refund is an *amount* against the PaymentIntent (refunds aren't line-item-aware), and one cart = N sibling `orders` rows sharing one PI (`webhook.ts:185` writes one row per product). So "refund the order" with no amount would silently refund the **whole cart**. The in-app refund takes an explicit `amount_cents` (defaulting to the clicked line) and flips/relists only the pieces the owner marks returned — the single-item case is just the trivial one-piece default.
- **Why `renderOrders` keeps no sibling stash.** `openRefundPanel` fetches the cart's full sibling set by PaymentIntent (Phase 1.1c + 1.5d), which is robust regardless of the active orders subtab — a multi-piece cart's siblings can straddle Needs-Shipping/Shipped, so grouping from the loaded subtab slice would silently under-list a partial cart. A `state.orders` stash would just be dead code beside the by-PI fetch.
- **Why the "no payment to refund" tag exists.** Healthy orders always carry a `stripe_payment_intent` from the webhook; the tag is belt-and-suspenders for old/imported rows missing one (opening the panel on one would just 409 at submit anyway — the handler guards it).
- **Why the button reads "Refund this purchase…"** A multi-piece cart shows N order cards that all share one PaymentIntent, and the panel lists every sibling — so "this purchase" matches what actually opens, where "Refund order" would mislead.

## WS2 — Coupons

- **Why human-readable dates (`expires_display`).** A raw Unix `expires_at` was once misread as July when the coupon was correctly set to June — the coupon was never wrong, the *display* was. The backend now returns `expires_display` alongside the raw value so neither surface ever decodes a timestamp.
- **Why the decode-free dollar strings (`min_display` / `amount_display`).** `min_amount` was settable on both surfaces but read back on neither — set a minimum and you couldn't confirm it from /admin *or* chat. And the GPT otherwise sees `min_amount: 5000` sitting next to a non-cents `percent_off: 20`, so it has to *infer* that one is cents — exactly the kind of inference that produces a 100×-off readback. The dollar strings (same pattern as `expires_display`) let it report dollars verbatim. `/admin` keeps rendering raw cents via `centsToDollars` — it already does the math correctly.
- **Why no Action response-schema edit.** Both coupon ops declare an **opaque 200** (`'200': { description: … }` with no `schema.properties`), so the platform passes the raw JSON body to the model — a new wire field is GPT-visible without any schema change. (Contrast `refundOrder`, which *does* enumerate its `relist` response — there the shape is declared, so additions there must be too.)
- **Deferred idea (optional stretch, not built):** `campaign_note` / `intended_dates` in the coupon `metadata` — a spot to record a sale's human intent. Noted from the original plan; out of scope for this build.

## WS3 — Chat-attach upload

- **Why a unique role per attached file.** The server names each R2 object `{role}-{slug}`, so two files resolving to the same role would silently overwrite each other (no error). A collision (a duplicate explicit role, or a positional fallback landing on an already-claimed role) is rejected with a per-file failure instead of destroying the earlier upload; the role is claimed only on a *successful* upload, so a failed file never blocks a later retry of its role.
- **Why attach is images-only (video rejected loudly).** A video routed through attach would get an image role (positional default `hero`) and the GPT would drop its url into `images[]` → the page renders a broken `<img src=.mp4>` and the clip never shows (`populateMedia` reads `media[]`, not `images[]`). The guard surfaces the misroute loudly even if the GPT slips past the instruction; video stays by-link (`uploadImage`).
- **Why pin `engines.node` to `">=20"`.** `handleAttachedRefs` uses `new File([bytes], …)` + `Buffer`, which are Node 20+ runtime globals — `tsc` can't catch a missing runtime global (`lib.dom` declares `File`), so leaning on Vercel's *silent* default runtime is borrowed time. Pin the floor as a documented contract.
- **Why keep `openaiFileIdRefs` as `items: { type: string }`.** That is OpenAI's documented Custom-GPT contract: the platform recognizes the property name and substitutes the string array with `{name,id,mime_type,download_link}` objects at runtime. Declaring the object shape in the schema would fight the documented substitution, and a stricter Action validator could reject a placeholder that no longer looks like a string array. The real-preview smoke test (TESTING item 14) is what proves the round-trip.
- **Why passing the correct role is load-bearing.** There is no stored role field — `upload.ts` names files `{role}-{slug}` and the frontend derives an image's role from that filename prefix. So the `roles[]`/positional mapping *is* the role; get it wrong and the piece lands in the wrong place on the page.
- **Why the `attachEventListeners` wiring is one consolidated block (and `add-media-row` lives in it, not 3.7b).** A missed init delta throws at init (`$('upload-btn')` is `null`, or a new wirer is forgotten) and **every handler registered after it** — orders subtabs, search, refund, coupons — silently never binds, so the build *looks* clean while whole capabilities are dead. A separate 3.7b `:152` edit + the WS4 block would both touch the same region and, in 1→6 build order, the 3.7b insert would drift the WS4 CURRENT anchor. Touching `:152-161` exactly once removes that hazard.

## WS4 — Admin polish

- **Why `#222`→`--c-accent` by role, not a blind find-replace.** The only two `#222` usages are `button.primary` and `.subtab-btn.active` *backgrounds* — both accent fills; body text is already `--c-text` via the global base. Role-matching guards a future `#222`-as-text from being mis-swept to the accent color.
- **Why the WS3.7 inline styles use the `var(--token, #hex)` fallback form.** Those styles build in WS3, *before* §4.1's `:root` lands, so the fallback makes them render correctly either way — and after the build the only hex left anywhere is inside such fallbacks (bare hex is the violation the item-24 grep catches).
- **Why `.label` is generalized.** Today's only `.label` rule is descendant-scoped `.order-info .label`, so the coupon code's `<span class="label">` in `#coupons-list` would render as plain inherited text. Dropping the prefix lets the order-card labels and the coupon code share one muted/uppercase affordance.
- **Why `productState` checks edits *before* sold.** A sold piece that also has staged copy edits returns `'edits'` (so it lists under **Live**, not **Sold**) because the unpublished staged edit is the actionable state the owner needs to act on. Flagged for render-tune — flip the precedence if a sold-first view ever reads better.
- **Why the pill semantics (sold≠red; refunded is its own class).** A sale is a *good* state, so `sold`→accent-soft, never a red error tint. `refunded` gets its **own** `.pill.refunded` class rather than reusing `.unsent`, which is semantically an *email-not-sent* state on order cards. `.pill` keeps `display:inline-block` so its padding renders.
- **Why the state legend has no "Refunded" swatch.** `productState` returns only live/edits/sold/draft/archived — no product card is ever `refunded` (that pill lives only on order cards). The `.pill.refunded` rule itself still exists for those.
- **Why `renderProductList` is byte-anchored.** The badge/filter/dim rewrite lands on code that *actively contradicts* it (the shipped fn builds two pills, reusing `.unsent` for "sold" and `.pill.shipped` for "live" — the exact semantics P2 removes). Left as prose, a builder could add the new badge *over* the surviving two-pill line and double the status.
- **Why `.media-row__head` must be a real class, not an inline `style=` grid.** An inline grid can't be overridden by the P5 media query, so the structured media editor would overflow at ~360px — the same columns bug fixed for `.img-url-row`. As a class it can collapse to one `1fr` column on a phone.
- **Why `openEditor`'s `#upload-status` reset must be repointed.** That reset line lives at the *end of `openEditor`*, not inside the deleted `onUploadImage` — so once P3d removes `#upload-status`, `$('upload-status')` resolves to `null` and `null.textContent` throws a `TypeError` on **every** editor open. It's valid syntax (invisible to `tsc`/`node --check`), runtime-only. Repointing it to clear `.upload-zone .zone-msg` preserves the clear-on-open.
- **Why the `seo_thumbnail` / `checkout_image` zones must set their column-backed fields.** Those two roles are *not* images-derived — `buildProductPayload` reads them from dedicated fields and the consumers read the **columns** (Stripe checkout image via `products.ts`→`stripeSync.ts`; the OG/social image via `main.js`/`product.js`). No code derives them from a role prefix (only `hero-` is derived, gallery is `gallery-`-filtered). A zone upload that only landed in `images[]` would leave the attribute unset — a silent miss on a money + SEO path.
- **Why the skeleton image block is `height:140px`.** It matches the real `.product-card img`; an `aspect-ratio:1/1` box was taller and reflowed the grid when skeletons resolved into real cards.
- **Why the P5 `@media` block must be physically last in `<style>`.** Source order ≠ fold order: P5 folds before P3, but if a builder appends the P3d base rules *after* the media query, equal specificity lets the desktop base win even at ≤640px and the mobile stack silently doesn't render.
- **Why "Copy address top-right" needs markup.** CSS alone can't reposition the after-the-`<pre>` sibling button, so the concrete default wraps a "Ship to" label + the button in a flex header; if that markup change is unwanted, "top-right" drops to render-tune and the load-bearing ghost-button restyle stays.
- **Why the focus ring broadens to all interactive controls.** Scoping it to inputs only left buttons/tabs/subtabs/links on the bare UA outline — uneven keyboard focus for a "polished + accessible" bar. The broadened rule uses `:focus-visible` only (no global `outline:none`), so any unstyled control keeps its UA fallback.

## WS5 — Homepage

- **Why build-time re-rendered MP4, not a runtime filter.** The HyperFrames runtime artifact (`@hyperframes/player`) *plays a composition* — it is not a real-time filter for an arbitrary site `<video>`, so "runtime via HyperFrames" doesn't exist. The alternative (a hand-rolled WebGL/CSS-grain shader) is fragile, costs per-frame on mobile, and fights the existing parallax transform. Build-time is deterministic (HyperFrames' core property), costs the browser exactly one `<video>` decode (today's cost), and is a drop-in asset swap that preserves every existing hero layer.
- **Why `color:transparent` on the real `<h1>`, not an absolute-clip.** The old absolute-clip idiom took the `<h1>` out of flow; since `.hero__title-lottie` is also `position:absolute`, the wrapper (no explicit height) collapsed to 0 → the Lottie's `inset:0` box rendered at height 0 → the write-on was invisible in its *success* path while every fallback looked fine. `color:transparent` keeps the `<h1>` in flow (defining the box height) and in the a11y/DOM tree for SEO/SR; the Lottie draws on top.
- **Why a versioned CDN key, not an in-place overwrite.** The hero objects are served `immutable, max-age=1yr`, so overwriting the same key won't bust caches — a `-v2` key (or a Cloudflare purge) is required for any static-asset update.

## Phase 3.9 — GPT instructions (full rewrite)

- **Why the file was rewritten whole, not patched section-by-section.** The 8000-char cap can't be met by patching: the WS1–3 rule adds take the existing file from ~7781 to ~8921, and *every* rule-preserving trim across the whole file bottoms out around 8538 — still well over. So it was rewritten via a 3-pass copywriting method (cut-down → restructure → remove-redundancy): dense run-ons broken into clean labeled lines (which the GPT also reads better), cross-section redundancy removed, every distinct rule kept. Result: `wc -c` = 7788 / 8000. The cap is HARD — any future instruction edit must re-run `wc -c`.

## WS6 — Inventory

- **Why stock lives only in `products.quantity` (Supabase), never Stripe.** Stripe Products/Prices have *no* stock field — you pass a *transactional* line-item `quantity` per Checkout and Stripe forgets it after charging. Stripe's own limited-inventory guidance is to keep stock in your DB and decrement on `checkout.session.completed`. The pre-WS6 webhook flipped `available=false` on *any* sale and never touched `quantity`, so a multi-stock piece was stranded after one sale.
- **Why the cutover `quantity=0` UPDATE ships COMMENTED.** A blind apply would wrongly zero an intentionally-paused-but-previously-sold piece in a template "User"'s multi-stock catalog. A doc-only "remember to eyeball" note can't bind the operator — shipping the destructive statement commented means `supabase db push` applies only the safe `record_sale` function, and the operator runs the SELECT, confirms each listed row is genuinely sold, then uncomments + runs the UPDATE by hand, once. The gate is mechanism, not memory.
- **Why `record_sale` uses `unnest`/`count(*)` instead of a flat `- 1`.** `p_ids` has no duplicates today (the cart dedupes by `product_id`, each line-item is hardcoded `quantity: 1`, and a piece sells to separate sequential orders — never N-in-one-cart), so `- 1` would also be correct. The grouped-count form is zero-cost hardening: it's unambiguously right and future-proofs a quantity selector if one is ever added.
