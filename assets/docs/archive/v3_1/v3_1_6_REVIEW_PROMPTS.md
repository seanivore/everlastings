## The settled base — don't re-litigate what's shipped

The current store — everything described in `EVERLASTINGS_STORE.md` and present in the repo today — is **built, tested, and approved to go live**; it's the fixed, proven substrate. **v3.1.6 is a delta layered on top** (refunds, coupons-in-admin, chat-attach upload, admin polish, homepage). Reviewers judge the **v3.1.6 changes** and whether they **fit** that base — they do **not** redesign or re-flag settled/shipped behavior, and a finding must be about what *this build* adds/changes (or a real conflict it creates with the base). This is inlined into every block so reviewers don't manufacture blockers out of proven code.

## ⭐ The review lens (in every block — both parts)

**(a) North Star / thesis:** minimize Em's friction to manage her store — **every management capability must be doable in BOTH the /admin panel AND her Custom GPT, equally** (we can't rely on the GPT always being there). The GPT should do what a capable agent (Claude Code + skills/MCPs) could. A capability that reads "covered" in the docs but that Em can't actually trigger (in either surface) is a **real gap**, not a nitpick.

**(b) Broader mandate — the lens must NOT shrink the scope:** the North Star is the primary *functionality* lens, but it is **not** the filter for what counts as a gap. Across the **whole build (IMPLEMENT + BOTH addenda), every element**, also catch anything not truly **exclusively-executable** (the builder would have to discover/decide), any **unvalidated assumption**, and any **design-correctness** failure (the "columns-bug" class: a spec that *applies* cleanly but *renders wrong/incomplete*). Do not neglect the design addendum because the functionality lens feels primary — that exact mistake dropped the design review in v1.6's first round.

**(c) Read in full, co-design (don't just audit), flag-don't-assert.** Read ALL provided docs **end-to-end** — don't ration tokens (you're at maximum effort and context is managed for you; the four docs + STORE total ~48k tokens; grep/jump reading produced real mis-diagnoses last round). Think comprehensively: within the v3.1.6 delta, **a gap is a gap whether it's a flaw in what we wrote OR something the delta should address but omitted** — don't assume the build is mostly complete and you're only auditing what's present (distinct from the settled base: don't redesign the shipped substrate, but DO surface what the delta itself should cover). And you often can't see runtime/code behavior — when a finding depends on it, **FLAG it as needs-verification, do NOT assert it as broken** (in prior rounds several "broken" findings were contradicted by the actual code — round-2's "critical data-corruption", and round-3's "an undeclared response field is invisible" (the coupon ops are opaque — description-only 200s — so the GPT reads raw JSON), "the chat-attach branch is unauthenticated" (auth runs at upload.ts POST top), and "the hero CSS tokens are missing" (they exist) — though note the round-3 multi-cart over-refund WAS real, so flag-don't-assert cuts both ways — and round-5 added THREE more code-contradicted false alarms: "#222 must be body text" (it's the accent-button + active-subtab background), "homepage.js may load synchronously" (it's `defer`), ".hero h1 may be a direct-child selector" (it's a descendant) — all now verified and recorded in the docs, so for a delta on a settled base prefer "I can't verify X from the docs" over "X is broken").

## Landmines — validate against reality, not training data (inlined into each block)

1. **Vercel `api/*.ts` must compile to CommonJS** (the project `tsconfig`); ES-module output crashes the deployed runtime and **only surfaces on a real preview URL** (masked by `vercel dev`). The new `orders.ts` `POST` + the `upload.ts` changes must stay tsc-clean AND CJS-safe.
2. **No new Vercel function** — refund folds into `api/orders.ts` (new `POST` + a `vercel.json` rewrite); chat-attach folds into `api/upload.ts` (a rewrite). `ls api/*.ts` count unchanged (Hobby cap 11/12).
3. **`is_test` is never user-editable;** every new read/write stays scoped to `isTest` (`api/_lib/env.ts`). The refund order-lookup is `is_test`-scoped; Stripe's secret key is env-scoped (test vs live).
4. **The frontend derives an image's ROLE from its filename prefix** — `product.js:415` (`/\/(?:test_)?gallery-/`) + `:576` (`hero-`). There is **no stored role field**: the server names files `{role}-{slug}.{ext}` from the `role` param, and the page reads role back off that name. So passing the **correct role** is load-bearing; a wrong/missing role silently mis-categorizes or drops an image. Nobody renames a file.
5. **`openaiFileIdRefs` is documented but finicky** (blank arrays / 422s in the wild) — the by-link `uploadImage` is the **deliberate backstop** and must stay. `files.oaiusercontent.com` is public https → passes the existing `isPublicHttpUrl`; `download_link`s expire ~5 min; ≤10 files/call.
6. **The GPT Instructions have an ~8k-char cap** — the v3.1 edits (refund flip, attach-first, alt-text, coupon read-back, poster aside) must fit; flag if the NEW text plausibly blows the budget.
7. **Refund is AMOUNT-BASED** against the PaymentIntent (Stripe refunds aren't line-item-aware), and **one cart = N sibling `orders` rows** sharing one `stripe_payment_intent`. `refundOrder` refunds `amount_cents` (default = the clicked order's line amount) and flips/relists **only** the marked `relist_product_ids` — never the whole cart. `idempotencyKey = refund-<pi>-<amount>-<ids>` (a double-click is idempotent; a different piece/amount on the same PI isn't blocked). The `charge.refunded` webhook (`api/webhook.ts:60-89`) flips `orders.status='refunded'` on a **full-PI** refund (idempotent with the handler's per-order flip); an intermediate partial doesn't fire, so the handler owns that piece's flip. Don't introduce a double-write hazard, and don't re-flag the per-order button as over-refunding (round-3 headline — fixed).
8. **Refund never auto-relists** (a damaged-item refund mustn't silently re-list). Relist is a separate, **state-aware**, confirmed step (`available:true` if published-but-sold; `unarchive` if archived). **The refund's `products(...)` Supabase embed must deserialize to a to-one OBJECT** (read like the GET at `orders.ts:65`/`:172`) — if it's wrong the refund still succeeds but `relist` is malformed and the "put it back up for sale?" prompt **silently never fires** (a parity hole `tsc` won't catch — verify against a real response).
9. **The /admin upload control already uses the SAME `/api/upload` pipeline** (multipart → Cloudinary per-role crop → R2/CDN). The authorized admin `GET /api/products` returns **full** rows, so `stripe_product_id` / `is_published` / `archived_at` are present for the coupon scope picker + product-state filter.
10. **Coupons: never make the GPT decode a raw Unix timestamp** (the `FEEDBACK_COUPON_v2_1_0` regression — it misread July for June). The backend returns `expires_display` (human, store TZ America/New_York) **alongside** `expires_at`; the GPT relays that. `product_ids` are **`stripe_product_id`** (NOT the Supabase id); a product-scoped coupon needs a **published** product.
11. **The structured admin media editor must emit the SAME shape** `product.js`'s `populateMedia` reads — `{ type:'video'|'youtube', url, autoplay?, loop?, poster?, alt? }` — so `api/products` + `product.js` stay untouched; MP4s render before YouTube; empty media hides the section.
12. **`cleanUrls: true`** — `vercel.json` rewrite *destinations* must drop `.html` (silent 404s otherwise). The two new rewrites are API→API (no `.html`), but confirm.
13. **Storefront brand is untouched.** /admin gets a **neutral/template** aesthetic (the addendum's `:root` tokens) — NOT the Everlastings plum/lavender/serif, and not anchored to august.style tokens. The token literal-sweep must not leak storefront brand into /admin (or vice-versa).
14. **Reduced-motion + a11y preserved.** The hero's `prefers-reduced-motion` fallback (`styles.css:376`) stays; the new Lottie title is `aria-hidden` decoration over a **real `<h1>`** (SEO + screen-reader + no-JS + reduced-motion fallback to the static title).
15. **The hero CDN objects are served `immutable, max-age=1yr`** — the old-film hero swap needs a **versioned key** (e.g. `homepage-hero-animation-v2.mp4`) + URL edits, never an in-place overwrite (cache won't bust).
16. **The go-live (`main`) version is untouched** by v3.1.6 — it's a separate `dev` cycle that ships on its own later. Don't entangle them.
17. **WS4/WS5 are spec'd as executable DESIGN (concrete default + render-tune), not byte-anchored like WS1–3.** Judge the design addendum by "concrete enough that the builder never guesses + Sean can render-tune on the live preview," not by final-pixel values. Real content (the Lottie JSON, the HyperFrames render) is a content-creation step, never a build/test gate. The two remaining mechanical bits flagged in WS4 (the token literal-sweep + P0's product-list state-filter JS / back-nav) must be concrete enough to apply without deciding.
18. **Inventory lives ONLY in Supabase `products.quantity`; Stripe holds NO stock** (the Checkout line-item `quantity` is transactional, forgotten after charging). **WS6** makes a sale **decrement** `products.quantity` (the atomic `record_sale` RPC) + set `available=(quantity>0)`; it never touches Stripe and **never archives**. Don't assume a sale zeroes quantity, deactivates a Price, or that Stripe tracks stock.
19. **Refund relist restores BOTH axes (WS6.3):** unarchive when archived AND `quantity+1` (which re-flips `available` true) — not XOR; a relisted piece must be genuinely **buyable** again.
20. **The assembled GPT instructions must stay < 8000 chars** after all WS1-3 edits (the static gate checks it) — over the cap the GPT silently truncates its instructions.
21. **ONE canonical `productState(p)`** (from `is_published`/`draft`/`available`/`archived_at`) drives BOTH the P0 filter tabs AND the P2 status badge — no divergent predicates.
22. **The architecture doc was just reconciled, but a doc line can still lag the code** (last round tripped on a stale ":678 quantity=0" line the webhook never did) — verify behavior against the build guide + code, flag-don't-assert, and treat the doc's Status/Version header as suspect first.
23. **Full-capability parity — both directions, no single point of failure (Sean's standing rule).** /admin must do EVERYTHING the GPT can AND vice-versa; never leave the owner dependent on the GPT (it can be down). Design for the reusable-template **"User"** with a large catalog, NOT Emy's tiny one — optimizing for "how Emy will use it" is exactly what made us double back before (e.g. by-link share vs. a real upload). A capability present in one surface but not the other is a **real gap** (this round closed the coupon multi-product scope in /admin to match the GPT).
24. **The cart is one-of-each-per-order.** It dedupes by `product_id` (`main.js:121` returns early on a re-add), there is no quantity selector, and each checkout line-item is `quantity:1` — so `metadata.items` / `productIds` / `record_sale`'s `p_ids` **never contain a duplicate id**. A qty-N piece sells via N **separate sequential orders**, not N-in-one-cart. Don't read `record_sale`'s per-row decrement as an under-count bug (the round-2 headline finding did — it can't fire under this cart).
25. **The refund relist prompt ALWAYS offers now** (Sean's call — minimize friction, never strand stock): the wording branches on current state — a **down** piece (sold-out or archived) → "re-list + make available again", an **in-stock multi-qty** piece → "+1 to its available quantity"; both restore the returned unit (`quantity+1`, unarchive if archived). It still never *silently* relists — it asks.
26. **One PaymentIntent spans N sibling `orders` rows** (`webhook.ts:185` writes one row per product) — a refund is an **amount** against the PI, not per-row. The round-3 build's amount-based composer passes `amount_cents` + `relist_product_ids` so it refunds/relists only the **marked** pieces (the per-order Refund button used to refund the whole cart — that was the round-3 headline, now fixed). Don't re-flag the button as over-refunding, and don't mis-read the one-PI/N-orders fan-out as a bug.
27. **The coupon Action responses are OPAQUE** — `listCoupons`/`createCoupon` declare a 200 with a *description only* (no `schema.properties`), so the GPT receives the **raw JSON** and a new wire field (e.g. `expires_display`) is visible **without** a schema edit. (Contrast `refundOrder`, which DOES enumerate its `relist` response — additions there must be declared.) Don't flag "undeclared response field is invisible" for the opaque coupon ops.
28. **Auth is at the top of `upload.ts` POST** (`:118`) — `authorize(request)` runs before the JSON content-type fork that reaches `handleAttachedRefs`, so the chat-attach branch is already gated. Don't flag it as unauthenticated.
29. **The GPT-instruction file was FULLY RESTRUCTURED this round (round-4 #1) to fit the hard 8000 cap — Phase 3.9, verified 7651/8000.** Review the instructions against **Phase 3.9's final text**, NOT the per-section 1.4/2.3/3.5 CURRENT→NEW blocks (those document the rule deltas; the file ships wholesale from 3.9). The cap is HARD — any instruction add must re-verify `wc -c`. Don't re-flag the cap as unmet (it's closed) or critique the superseded dense wording.
30. **Round-4 fixes already landed — DON'T re-flag:** `endOfDayET` uses `formatToParts`/`shortOffset` (not a brittle `new Date(localeString)` round-trip); the WS6 cutover update is scoped to actually-sold rows (`exists (orders)` — won't zero a paused piece's stock); every `.pill` state has concrete CSS incl. `.pill.refunded`; `/admin` upload nudges alt text (GPT-parity); the Lottie init has a `data_failed` + SVG-content guard (no blank hero). STORE's pre-WS6 "no decrement" lines (`:614`/`:709`) are correct **shipped** truth synced at the as-built Phase 6.4 — a mid-build STORE edit was proposed and **rejected** (no-mixed-truth); don't re-flag it as a contradiction. The /admin refund panel fetches a cart's **full** sibling set via `GET /api/orders?payment_intent=` (round-4 breadth fix, Phase 1.1c — NOT the active-subtab slice), so don't flag it as under-listing a partially-shipped multi-piece cart.
31. **The GPT schema file has a per-`summary`/`description` 300-char cap** (the static gate checks `every summary < 300`) — as hard as the 8000 instruction cap. COUNT every new/edited schema string; don't estimate (round-5 #1 caught a `refundOrder` summary at **314** the doc had estimated "≈295" — it would have failed the gate on minute one).
32. **Attached VIDEO routes by-link only** (`uploadImage`), never `uploadImages` — enforced in BOTH the MEDIA instruction AND a server guard in `handleAttachedRefs` (rejects `video/*` with a "send it as a link" failure). An attached video would otherwise get an image role + land in `images[]` and break the page (`populateMedia` reads `media[]`, not `images[]`). Round-5 #2 — don't re-flag.
33. **Round-5 verified-against-code, now recorded — DON'T re-raise as risks:** the two `#222` literals are accent backgrounds (→`--c-accent`; body text is already `--c-text`); `homepage.js` is loaded `defer` (so the Lottie defer-ordering holds); `.hero h1` is a **descendant** selector (so the `.hero__title` wrap keeps it matching).
