**Scope reviewed:** `v3_1_0_IMPLEMENT.md` + `v3_1_0_ADDENDUM_DESIGN.md` + `v3_1_0_ADDENDUM_TESTING.md`, judged against `EVERLASTINGS_STORE.md` (the settled base) and the 17 accumulated landmines — not against doc-internal consistency.

Verdict: NEEDS ANOTHER PASS. 

Nothing here is a compile blocker — the WS1–3 byte-anchoring, is_test/idempotency/CJS discipline, and the homepage a11y/reduced-motion work are solid. But two of the findings ship a feature that looks done and isn't, and a few load-bearing claims aren't anchored to their own standard.
The one thing: the refund→relist loop was never validated against the real order-status and buyability models, and it has two silent dead-ends the spec and the tests miss:

F1 — you probably can't reach the order to refund it. The base describes the admin Orders view as the needs-shipping queue (status='completed' AND shipped_at IS NULL), and the plan never quotes GET /api/orders or the orders subtabs. The common refund is a shipped/delivered order — not in that queue — so in both surfaces you may have no way to locate its id. And post-refund the 'refunded' order drops out of a status-scoped view, so the "Refunded" pill (1.5a) is dead code and testing item 1's assertion can't hold.
F2 — even when you relist, it isn't buyable. The purchase webhook sets available=false and quantity=0; relist sends only {available:true}, so quantity stays 0 → Add-to-Cart stays gated (quantity>0). The piece shows "available" but can't be bought. Item 1 checks "shows available," not "is purchasable" — so it passes green while the payoff is broken. (Plus archived+sold is treated as XOR, when a piece can be both.)

The rest, ranked, in the doc: F3 the coupon list/create field contract isn't anchored (silent wrong-render, not a crash); F4 the structured MP4 editor drops controls — a click-to-play correctness and admin↔GPT parity gap, and it contradicts the plan's "same shape" claim; F5 the ~8k GPT-instruction cap is unguarded after your WS1/2/3 rewrites; F6 admin has no video upload role; F7 P0's filter tabs and P2's pills don't share a defined state taxonomy; then F8–F12 (service-role embed for archived relist, coupon-expiry TZ drift, WS5 not byte-anchored incl. hardcoded hero copy, Stripe-config-gated refund email, and doc-accuracy nits). All 17 landmines validated — 6, 8, 10, 11, 17 are where the gaps surface.
Fix F1–F2, anchor F3, close F4+F5, and it's buildable.

## Ranked findings

### F1 — Refund is likely **unreachable for the orders you'd actually refund**, in BOTH surfaces (and the post-refund state may be invisible)
**Where:** WS1 (IMPLEMENT 1.5a/1.5c) + schema `refundOrder.id` "from listOrders" (1.3); against `EVERLASTINGS_STORE.md:599` ("lists orders by shipping status"), `:319` ("/admin/orders shows 'Needs shipping' queue"), `:1057` (partial index `WHERE shipped_at IS NULL AND status='completed'`); TESTING items 1–2.
**What's wrong:** The plan adds the Refund button inside `buildOrderCard`, so it appears on *whatever cards render* — but never anchors **what `GET /api/orders` returns** nor **the orders-subtab statuses** (`admin/index.html:243-256`, referenced but never quoted). The base describes the admin Orders view as the *needs-shipping queue* (`status='completed' AND shipped_at IS NULL`). The most common refund is a **shipped/delivered** order (buyer received it, it's damaged) — which is *not* in that queue. Two concrete dead-ends follow:
- **(a) Entry point:** if the orders list / `listOrders` only surface unshipped-completed orders, neither /admin nor the GPT can locate a shipped/delivered order's id to refund it. The refund capability reads "covered" but isn't drivable for the primary case.
- **(b) Post-refund visibility:** the action flips `status` to `'refunded'`; `submitRefund` then calls `setTimeout(loadOrders, 800)`. If the reloaded view is status-scoped, the just-refunded order **drops out** — so the "Refunded" pill (1.5a) is dead code and TESTING item 1's assertion ("the order flips to the Refunded pill") cannot hold.
**Fix:** Quote `GET /api/orders`' actual query + the orders-subtab definitions. Confirm (or add) a view that surfaces **shipped / delivered / refunded** orders (e.g. an "All" subtab) and a `listOrders` status filter so the GPT can fetch their ids. Add a test: *refund a **delivered** order from both surfaces, then confirm it appears under a Refunded/All view.* This is the highest derail/missing-capability risk in the build.

### F2 — Relist flips `available` but **doesn't restore buyability** (purchase zeroes `quantity`); archived+sold is a missed second axis
**Where:** IMPLEMENT 1.5c `relistPiece` (`PUT {available:true}` / `unarchiveProduct`), GPT instruction 1.4a, handler `relist` (1.1b); against `EVERLASTINGS_STORE.md:678` (purchase sets `available=false, quantity=0`), `:687` (buyable = `available=true, quantity>0`), `:388` (unarchive reverses `archived_at` + Stripe `active`, not `available`/`quantity`).
**What's wrong:** A one-of-a-kind sells → webhook sets `available=false` **and** `quantity=0`. Relist sends only `{available:true}` → `quantity` stays `0` → per the documented available-state (`quantity>0`), **Add-to-Cart stays disabled**: the piece shows "available" but is **not purchasable**. Worse, the two relist axes are treated as XOR: a piece that is *both* archived *and* sold (refund of an archived-then-sold piece) gets `unarchiveProduct` only → un-archived but still `available=false, quantity=0`. The "put it back up for sale" payoff silently fails for the common case. TESTING item 1 only checks "shows available again on /shop" — not buyability — so the test inherits the same blind spot and would pass green.
**Fix:** Define relist as *restore full buyability*: set `available:true` **and** `quantity:1` (or prior stock), **and** unarchive when archived (handle both axes, not one). Update the handler `relist` to expose `quantity`, `relistPiece` to `PUT {available:true, quantity:1}` (plus unarchive when `archived`), and the GPT instruction to match. Change the test to assert a **completed checkout** on the relisted piece, not just the listing state. *(Note: the byte-anchored `available:true`-only relist mirrors the settled advisory instruction — but this build automates it, so the latent quantity bug now ships as a one-click action.)*

### F3 — Coupon endpoint I/O contract is **not anchored** — the admin list/create depend on field names that aren't in any quoted block
**Where:** WS2 `renderCoupons` / `onCreateCoupon` (IMPLEMENT 2.1f); only `products.ts:786` (`expires_at: pc.expires_at ?? null`) is quoted from the list-return object.
**What's wrong:** `renderCoupons` reads `c.percent_off`, `c.amount_off`, `c.store_wide`, `c.product_ids`, `c.times_redeemed`, `c.max_redemptions`, `c.code` — **none anchored**. If the existing `handleCouponList` shapes any of these differently (e.g. nested scope, `discount_*` names), the list silently renders "*discount · store-wide · 0 used*" for everything — it won't crash, so tsc + a glance both pass. Same for **create**: `onCreateCoupon` posts `{type, value, code?, min_amount?, max_redemptions?, expires_at?, product_ids?}` with `value` = percent number for percent but **cents** for amount — the endpoint's actual field names + the percent-vs-cents `value` semantics are stated in prose (locked-decisions :54) but never quoted from `handleCoupon` (`products.ts:733/689-798`). A cold builder cannot verify either without opening files the plan didn't quote.
**Fix:** Byte-anchor the **full current per-coupon return object** of `handleCouponList` and the **request-parsing** of `handleCoupon` (the accepted field names + the `value` unit semantics). The plan's own standard is "quote the verified current block as the anchor for every load-bearing claim" — these two are load-bearing and unquoted.

### F4 — Structured MP4 editor omits `controls` — breaks click-to-play render AND admin↔GPT parity
**Where:** IMPLEMENT 3.7b `collectMedia`; DESIGN P3c; against `EVERLASTINGS_STORE.md:1002` (media shape is `{type, url, alt, loop, autoplay, controls, poster}`).
**What's wrong:** `collectMedia` emits only `{type, url, autoplay?, loop?, poster?, alt?}`. The documented shape — and the field the GPT can set ("click-to-play with sound (the default)… set the media flags accordingly") — includes **`controls`**. The editor's only video state is the "Autoplay + loop, silent (GIF-like)" checkbox; **unchecked** emits `{type:'video', url}` with no `controls`. If `populateMedia` renders a play affordance for a non-autoplay clip *only when `controls:true` is present*, every click-to-play clip added in /admin renders **unplayable**. Either way it's a **parity gap**: the GPT (and the old raw-JSON textarea) can set `controls`; the structured editor can't. The plan's claim that the editor emits "the SAME shape `populateMedia` reads" is false — it emits a strict subset missing exactly the field that makes click-to-play work.
**Fix:** Verify `populateMedia`'s handling of a non-autoplay video without `controls`. If a play control needs `controls:true`, have `collectMedia` emit `{type:'video', url, controls:true}` on the non-autoplay branch (and read it back in `addMediaRow`). Confirm against `product.js`, since landmine 11's stated shape and the base's documented shape disagree on `controls`.

### F6 — Admin can't upload a video with a proper **video role** — admin↔GPT video parity + TESTING item 23
**Where:** IMPLEMENT 3.7c (only the `skip_transform` auto-infer is added); TESTING item 23 ("upload an MP4 via the admin control").
**What's wrong:** 3.7c makes a video MIME auto-skip the Cloudinary crop, but the plan never extends the `#upload-role` select to offer a **video-0N** role. So an MP4 uploaded via /admin gets named under whatever image role is selected (e.g. `gallery-0N-slug.mp4`), not `video-0N`. The structured editor's `m-url` is a free paste field, so it *works* if Em pastes that URL — but the "upload an MP4 via the admin control" path (item 23) has no clean role, and admin doesn't reach the GPT's `video-0N` convention.
**Fix:** Add a video role option (`video-01..05`) to the admin `#upload-role` select so an admin-uploaded MP4 is named + transform-skipped consistently and is selectable in the MP4 editor. Anchor the current `#upload-role` `<option>` list (not quoted today).

### F7 — Product-state taxonomy is **undefined** — P0 filter tabs and P2 pills don't share a model
**Where:** DESIGN P0(b) (tabs: All / Live / Draft / Sold / Archived) vs P2 (pills: draft / edits / live / archived / sold / available); IMPLEMENT 1012 lists "P0 state-filter JS" as mechanical-from-spec; landmine 17 (this is a flagged mechanical bit that "must be concrete enough to apply").
**What's wrong:** P2 renders **six** states (incl. an **"edits"** pill = published row with a non-null `draft`, and **"available"** distinct from **"live"**); P0 offers **five** tabs with **no "Edits" filter** and an undefined **Live vs Available** distinction. The label→predicate mapping is never specified, and the predicates overlap (is a published, not-archived, sold piece "Live" or "Sold"? both?). The builder must invent the predicates — exactly the decision the plan is supposed to make.
**Fix:** Define the canonical taxonomy once, from `is_published` / `draft` / `available` / `archived_at`, e.g.: *Archived* = `archived_at != null`; *Draft* = `!is_published && !archived`; *Edits* = `is_published && draft != null && !archived`; *Live* = `is_published && available && !archived`; *Sold* = `is_published && !available && !archived`; *All* = everything. Have **both** P0 tabs and P2 pills derive from this one function (and decide whether "Edits" gets a tab).

### F8 — `requireAdmin`'s client must be **service-role** or relist-from-archived silently no-ops (untested)
**Where:** IMPLEMENT 1.1b refund query `products(id, slug, title, available, archived_at)`; against `EVERLASTINGS_STORE.md:383` (admin/GPT read via service-role, bypassing the `archived_at IS NULL` RLS); TESTING item 2.
**What's wrong:** The plan flags the embed *shape* (to-one object, landmine 8) but not its **RLS visibility**. The existing `GET /api/orders` reads `order.products?.title` only for needs-shipping-queue orders, whose products are live/sold — it has **not** proven the embed resolves for an **archived** product. If `requireAdmin` ever returns an RLS-bound client, the embed returns `null` for an archived piece → `relist` is `null` → the unarchive prompt never fires. Item 2 (refund an archived piece → unarchive) is the first time this path is exercised.
**Fix:** Confirm `requireAdmin` returns the service-role client (per :383). One line, but newly load-bearing and currently untested for the archived case.

### F9 — Coupon expiry is built in the **browser's** local end-of-day but displayed in **store TZ** — latent date drift
**Where:** IMPLEMENT 2.1f `onCreateCoupon` (`new Date(`${expires}T23:59:59`)` → browser-local) vs 2.2a `formatExpiry` (America/New_York); the whole point of WS2's date work (landmine 10, the FEEDBACK_COUPON regression).
**What's wrong:** The stored `expires_at` is end-of-day in the **owner's browser TZ**, but the displayed/relayed value is **America/New_York**. For an owner not in Eastern, the picked calendar date and the effective end-date can land a day apart (e.g. `2026-06-21T23:59:59` Pacific → ~`Jun 22, 3 AM EDT`), and the create-confirm shows the raw input date while the success toast shows the store-TZ date — the exact misread class WS2 set out to kill. Em is in Massachusetts (Eastern), so it's correct *for this store today*; it's a latent bug against the "ports to any client" goal and an owner who travels.
**Fix:** Construct the expiry as end-of-day in **America/New_York** (TZ-aware), so input date, stored `expires_at`, and `expires_display` all agree regardless of the owner's locale. (Low practical priority for Em; flagged because it's the precise failure WS2 exists to prevent.)

### F11 — Refund buyer-email + `charge.refunded` reconciliation are **Stripe-config-gated** and can't be verified in test mode as written
**Where:** schema `refundOrder.summary` ("emails the buyer automatically"); TESTING items 1 + 6; against `EVERLASTINGS_STORE.md:918` (Stripe customer-email "Refunds" toggle is a setup step) and `:362/:806` (charge.refunded reflection in `webhook.ts`).
**What's wrong:** The buyer refund email is **Stripe-sent**, gated on the dashboard "Refunds" customer-email toggle — and Stripe generally doesn't deliver customer emails in **test mode**, so item 1 can't actually confirm "buyer emailed" on the preview. **Fix:** In the TESTING env prerequisites, (1) note the buyer email is Stripe-sent/live-gated and don't assert delivery in test mode; (2) require the preview Stripe webhook endpoint to include `charge.refunded`. No code change.
