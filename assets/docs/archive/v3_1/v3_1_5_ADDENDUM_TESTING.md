## WS1 — Refund (both surfaces — amount-based)

> **🎯 Headline — multi-item cart, no over-refund (round-3).** Build a cart of **3 different pieces** (one checkout → 3 sibling `orders` rows sharing one `stripe_payment_intent`) and complete a test purchase. On any one order's card → **Refund order** → the panel lists **all 3 pieces** with their charges → check **one** → the amount fills to that piece's charge → Refund → in the Stripe test dashboard the refund is **only that line's amount, NOT the cart total**; **only that piece** flips `refunded` + relists; the **other two stay `completed` + available** + still buyable. Then check **two** → both refund + relist. Then **edit the amount down** (a restocking deduction) with a piece still checked → the smaller amount refunds, the checked piece still relists. *(A Stripe refund is an amount against the PaymentIntent and one cart is N orders — the old `{ payment_intent }`-with-no-amount call would have silently refunded the whole cart.)* **Sub-subtab independence (round-4 breadth-pass fix):** the panel **fetches the cart's full sibling set by PaymentIntent** (`GET /api/orders?payment_intent=…`), so **mark ONE of the 3 pieces shipped first** (so the siblings straddle Needs-Shipping/Shipped), then open Refund **from the default Needs Shipping subtab** → the panel STILL lists all 3 pieces (not just the unshipped ones). It must not silently under-list a partially-shipped cart from any subtab.

1. **/admin, published-but-sold piece — from the All Orders / Shipped subtab** (a refunded order drops out of Needs Shipping's `status=completed` filter, so use a view where it persists — F1): a test order exists for it → Orders tab → **Refund order** → the **refund panel** opens showing the piece + its charge, **pre-checked**, amount **pre-filled** → Refund → confirm → "Refunded"; the order shows the **Refunded** pill (`.pill.refunded` — T3·3) and the relist prompt fires → accept → "Refunded + relisted"; the product is **available** again on `/shop` **and a real test checkout completes on it** (buyability restored, not just the listing — F2). Verify the refund + its amount in the Stripe test dashboard.
2. **/admin, archived piece (incl. archived-AND-sold):** refund (panel, piece checked) → relist prompt → accept → the piece **unarchives AND its stock is restored** (`quantity + 1`) so it's back on `/shop` **and buyable**; decline on another → stays archived/down. (Exercises the both-axes relist.)
3. **GPT — single piece:** "refund <buyer>'s order for <product>" → it **reads back** piece + amount + buyer and waits → confirm → `refundOrder {id}` → reports refunded + **always offers** to restore (wording by state — re-list a down piece, +1 an in-stock one) → "relist" → `editProduct {id:<product_id>, available:true, quantity: relist.quantity + 1}` (and/or `unarchiveProduct {id:<product_id>}`) runs.
   - **GPT — multi-piece (proves `relist[]` is well-formed — round-4 A #10):** on the 3-piece cart from the headline test, "refund <buyer>'s order — two of the three came back" → it confirms WHICH two + passes `amount_cents` (summed) + `relist_product_ids:[both ids]` → **assert it then fires TWO separate relist offers, one per `relist[]` entry**, each naming its own piece (not one merged prompt) → accept both → each piece relists with its own `{id:r.product_id}`. (If `relist[]` ever came back malformed the prompt would silently never fire — this is where that surfaces.)
4. **Idempotency (T2·6):** trigger the SAME refund twice fast (double-click / retry) → the Stripe dashboard shows **exactly one** refund for that amount on the payment intent; the second response is **409** (already-refunded) **or** a **200 with Stripe's idempotent same refund** — never a second distinct refund. (Key = `refund-<pi>-<amount>-<ids>`, so a **different** piece/amount on the same purchase is NOT blocked — verify a second, distinct per-line refund succeeds.)
5. **Guards (incl. amount):** refund a non-existent id → 404; an already-refunded order → 409; an order with no `payment_intent` → 409; an undeterminable amount → 400; an amount over the refundable balance → Stripe errors → **502** surfaced.
6. **Always-offer relist (F5), per returned piece:** a piece **still `available`** (multi-qty, units left) → *"Increase <title>'s available quantity by 1?"* → accept → `quantity` **+1** (decline → unchanged, no error); a **sold-out** piece → *"Re-list … and make it available for purchase again?"* → accept → quantity 0→1 + available true.
7. **Partial / goodwill (in-app now):** in the panel **check nothing** (or uncheck) and type a **smaller amount** → that amount refunds in Stripe, **no piece relists**, the order **status is unchanged** (nothing was returned). The "open in Stripe" deep-link is still present as a backstop.
   - **Full-amount goodwill — status DOES flip (round-4 A #5):** check nothing but type the **FULL purchase amount** → Stripe records a full refund → `charge.refunded` reports fully-refunded → the order **flips to `refunded`** (item 8) even though nothing was marked returned, and **no relist prompt fires** (no pieces). **Don't assert "status unchanged" here** — a full-amount refund is a full refund by definition. (The GPT instruction warns her of this before confirming; the piece stays sold/`available:false` since it wasn't marked returned.)
8. **Webhook reconciliation + `is_test`:** a **full-PI** refund's `charge.refunded` flips **every** sibling order on that PI to `refunded` (idempotent with the handler's optimistic flip); an **intermediate partial** doesn't fire fully-refunded, so the **handler** owns that piece's flip — no error either way. `is_test` scoping holds: on the test preview a prod order id is **not** found (404).

## WS6 — Inventory (stock decrement on a sale)

32. **One-of-a-kind (qty 1):** buy it → the webhook runs `record_sale` → `quantity` 1→0 and `available` flips false → shows "Sold". Identical to pre-WS6 behavior.
33. **Multi-stock (qty ≥ 2):** set a piece's quantity to 2 → buy one → `quantity` 2→1, **`available` stays true**, still listed + buyable; buy the second → `quantity`→0, `available`→false ("Sold"). Proves a sale no longer strands extra stock.
34. **Atomicity:** two near-simultaneous completions on the same qty-1 piece never drive quantity negative (`greatest(…,0)`); the second is a no-op at 0.
35. **A sale never archives / never touches Stripe:** after a sale, `archived_at` is unchanged and the Stripe Product/Price stay `active` (the piece is re-sellable with NO new Price — only our `available`/`quantity` gate it).
