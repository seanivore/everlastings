## Finding 1 — The auto-load claim is FALSE (real bug, blocks launch quality)

### What the docs claim

Three places assert that key context files auto-load:

- `v1_4_3_INIT_PROMPTS.md:3` — "auto-loaded context (`.claude/CLAUDE.md` → `.agent/AGENTS.md` → `assets/docs/EVERLASTINGS_STORE.md` and `.agent/DEV_RULES.md`)"
- `v1_4_3_INIT_PROMPTS.md:20, 69, 133` — "Do NOT re-read `EVERLASTINGS_STORE.md`, `DEV_RULES.md`, `AGENTS.md`, or `CLAUDE.md` — these are already in your auto-loaded context."
- `v1_4_3_A_IMPLEMENT.md:7-12`, `v1_4_3_B_IMPLEMENT.md:7-13`, `v1_4_3_C_IMPLEMENT.md:7-13` — "Pre-loaded Context (Do Not Re-Read)" listing those same files.
- `v1_4_3_IMPLEMENT.md:14` — "Architecture-level reference… lives in `EVERLASTINGS_STORE.md`, which is auto-loaded into every orchestrator session via `.claude/CLAUDE.md`."

### What's actually wired up

I traced the Claude Code import chain:

- `.claude/CLAUDE.md` (18 bytes, single line) → `@.agent/AGENTS.md` — that's the only import.
- `.agent/AGENTS.md` (42 lines) → **no `@` imports at all**.
- No `~/.claude/CLAUDE.md` exists at user level.
- No SessionStart hook injects these files.

So the chain is just `CLAUDE.md → AGENTS.md`. **Stop.**

`EVERLASTINGS_STORE.md`, `DEV_RULES.md`, and `BRAND.md` are NOT auto-loaded. Your suspicion is correct.

### Impact if launched as written

A fresh orchestrator session, told "do not re-read these — they're already loaded," would simply never read them. That means:

- **Track A** never reads `DEV_RULES.md` (commit conventions, branch policy referenced throughout the guide) and never reads `EVERLASTINGS_STORE.md` (the architectural primer the guide cites as "AR #N" throughout).
- **Track B** never reads `BRAND.md` — the init prompt at line 88 even says "BRAND.md is loaded; let it inform color choices, typography, voice…" — and the Verification Gate requires "BRAND.md voice + visual standards visibly applied." Without the file actually loaded, this gate is unmeetable.
- **Track C** never reads `EVERLASTINGS_STORE.md` or `BRAND.md` for any voice/copy work.

This is the single biggest risk in the current state.

### Recommended fix (Option A — make the claim true)

Add the imports to `.agent/AGENTS.md` so the chain actually loads them:

```markdown
@../assets/docs/EVERLASTINGS_STORE.md
@./DEV_RULES.md
@../assets/docs/BRAND.md
```

Total payload: ~130 KB across the three files (`EVERLASTINGS_STORE.md` 64 KB, `DEV_RULES.md` 43 KB, `BRAND.md` 23 KB). This is fine for Opus with 1M context — well under any practical budget — and matches what every track guide already says is loaded.

After adding, verify by starting a fresh test session in this repo and confirming the system prompt includes those files. (You can do this with a single `/clear` followed by asking the agent to quote a known line from each file.)

### Alternative fix (Option B — make the docs match reality)

If you'd rather not bloat the system prompt for non-track work, instead:
- Strike the "Pre-loaded Context (Do Not Re-Read)" sections from A/B/C guides.
- Replace with "Required Pre-Reads" that explicitly tell the orchestrator to read each file at session start (just like Track C already does for the sibling track guides at `v1_4_3_C_IMPLEMENT.md:14-22`).
- Update `v1_4_3_INIT_PROMPTS.md:20, 69, 133` to drop the "do not re-read" line.

**My recommendation**: Option A. The track guides cite "AR #N" inline throughout and assume the architectural primer is right there in context. Option B requires the orchestrator to re-read these files at session start, which costs the same context and adds an extra read step. Option A is one edit to one file and makes every existing claim accurate.

## Finding 2 — Coverage of the split is solid, with one moderate gap

The Explore agent compared the three track guides against the scope claimed in their preambles and the init prompts. Summary:

### Verified clean

- **Verification Gates** exist at the top of each track guide and are concrete/testable (A: `v1_4_3_A_IMPLEMENT.md:39-45`; B: `v1_4_3_B_IMPLEMENT.md:45-50`; C: `v1_4_3_C_IMPLEMENT.md:50-59`).
- **Cross-track handoffs**:
  - The 409 cart-recovery contract is fully documented in Track A's `/api/checkout/reserve.ts` section, and Track C's C2 has the matching consumer flow.
  - The `<!-- PLACEHOLDER: name -->` convention is documented in both B1 and re-stated in C1 with an identical grep pattern. C's first action (`grep -rn 'PLACEHOLDER' .`) will reliably find B's markers.
  - All status codes used by C (400, 409, 410, 500) are documented inline in A2.
- **Phase 0 reconciliation** in `v1_4_3_IMPLEMENT.md:67-92` is clear: 8 deferred items, each tagged with owner and where it picks up. Track A's "verify, do not re-execute" instruction matches.
- **All five referenced context files exist** at the paths the docs claim:
  - `assets/docs/EVERLASTINGS_STORE.md` (64 KB)
  - `assets/docs/BRAND.md` (23 KB)
  - `.agent/DEV_RULES.md` (43 KB)
  - `.agent/AGENTS.md` (3.4 KB)
  - `.claude/CLAUDE.md` (18 B)

### Gap — `/api/contact` underdocumented (moderate)

`v1_4_3_A_IMPLEMENT.md:5` lists 14 endpoints by name and includes `contact`. The other 13 each get a full `#### EndpointName — api/path.ts` block with description, code example, and checklist. **`/api/contact` does not.** It only shows up as a TODO line at `v1_4_3_A_IMPLEMENT.md:1360`: "Create `api/contact.ts` — similar pattern, stores in Supabase or sends email."

Why this matters: Track C will need to wire the contact form on `contact.html`. Without a documented request shape, response shape, error codes, or success format, the C orchestrator has to either invent the contract (drift risk) or stop and ask. Track A's verification gate ("All 14 API endpoints respond correctly to smoke tests") is also unmeetable as written for this endpoint.

**Fix**: Add a `#### Contact — api/contact.ts` section to Track A's A2, mirroring the Newsletter section's structure. Should be short — it's a Supabase insert + Resend notification — but it needs the contract spelled out.

### Minor — error-state count off by one

`v1_4_3_INIT_PROMPTS.md:99` says "all 24 error states from the inlined Error States Reference." `v1_4_3_B_IMPLEMENT.md` B6.5 (lines 542–655) actually contains 25 entries.

**Fix**: Change "24" to "25" in the init prompt. Or recount and reconcile.
